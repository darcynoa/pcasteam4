Q&A with Chris Capaldi

What are some entry points for someone who wants to interfere with the system?

  • Open-ended, come up with something creative for security.
  • No real definition for the approach.
  • He wants to see our take on the vulnerability and how we mitigate it.
  • Important to see where entry points have been exposed in the past. (where have previous cars' security been compromised in the past.)

What happens if we calculate that a collision can’t be avoided?

  • That is considered a system failure and ideally should never happen.
  • But if we do find ourselves in this scenario we should do everything we can to mitigate the collision.
  • Since this is a system failure we should do anything we can beforehand to avoid this scenario.

What other actions could the system take to avoid collisions other than breaking? the system?

  • The system could steer around obstacles.
  • Potentially the system identifies a second path that is safe.
  • This may be slightly outside the scope of this project.

What needs to happen for the system to activate?

  • We need to define this requirement ourselves.
  • The main goal of the system is to achieve zero pedestrian collisions.
  • Something else to consider is the braking deceleration experienced by passengers in the vehicle.

Should the passengers be able to deactivate the pedestrian collisions system?

  • People historically turn off systems they aren’t comfortable with.
  • If the system was flawless we shouldn’t ever have to turn it off but that’s never the case.
  • Sometimes the system can fail or be damaged. It's good to have a possible override if there is a safety concern.
  • A driver should be able to manually take over the car.
  • We need to consider what it takes for the driver to override the PCAS system and in what scenarios it’ll be used.

What happens if a pedestrian moves in front of the vehicle at a non-right angle?

  • This is out of the scope of this project.
  • Our system will just be focusing on 90-degree angles for pedestrians.
  • For this to be deployed we’d need a new system to account for this.
  • May include in the report that in the future we would need an algorithm that would eventually take into account the trajectory of the pedestrian.

What resource limitations are there for the system?

  • For bandwidth keep within the scope of a system that can be deployed in a car.
  • Look into standard automotive systems and how much power they can support. Do some industry research to find a benchmark for this project.
  • Do some industry research to find a benchmark for this project.

What does the car look like? (To what extent is the vehicle autonomous?)

  • We are looking at this as driver assistance, it is not a fully autonomous vehicle.

If the system activates what kind of notification is there?

  • We need to define the notification the driver receives.
  • We should think of something effective, safe, and not annoying for the user. Some suggestions were a potential chime or vibration of the seat.
  • We should think of something effective, safe, and not annoying for the user. Some suggestions were a potential chime or vibration of the seat.

How important is the importance of lost time?

  • Distance detection of the sensor for a pedestrian is 35 meters.
  • For the scope of this project, we can assume we have the assured distance.
  • We should think of the consequences in terms of lost time.

Is there anything we should do for places of reduced visibility?

  • For the scope of this project, we can assume we’ll have a guaranteed distance of 35 meters.
  • In the future, we may want to include a reduced visibility mode for potential additions in our writeup.
  • We need to consider and clarify added consequences of this system of collision rate and lost time.

What kinds of overrides are there for the system besides braking?

  • Make this as intuitive as possible for a driver to take over the system with a manual override.
  • We could potentially set it up so any input from humans would override the system or add an emergency button to start override.

Is our system the only system that uses the brake by wire actuator?

  • Yes

What is the scope of the sensor?

  • No super strict sensor specifications
  • Assume that in all of our scenarios the sensor will be well within its detection range for detecting pedestrians.

When does the vehicle enter the failed operational mode?

  • We want to determine the conditions that cause the vehicle to enter the failsafe operational mode.
  • Need to determine the impacts of different scenarios to determine which failsafe operational mode will be the safest option.

What procedure should take place when a security breach occurs?

  • We need to define what we deem as the safest mitigation to a security breach.
  • We need to decide if we potentially need to stop the car, only use manual mode, or potentially a combination of both.

How should the sensor work while the car is reversing?

  • This is outside of the scope.

How can weather and nighttime monitoring work to cause interference with the system?

  • This is outside the scope of the project.
  • Typically we would use a radar device for monitoring like this.
  • We would have to consider the environment around the radar and how that would affect it.

What happens if the vehicle is driving at a speed that isn’t defined within the given requirements?

  • This is outside the scope of the project.

Any recommendations for minimizing lost time?

  • Completely up to us but we also need to minimize collisions.

How should the system react when there’s a pedestrian but no collision?

  • Analyze the risk associated with a pedestrian.
  • Consider if the pedestrian is a safe enough distance where they won’t need any action by the PCAS.

How close can the car come to the pedestrian and how should passenger comfort play into this distance?

  • It is generally up to us to define this distance there isn’t any rigid requirement.
  • Some things to consider are the comfort of the pedestrian and driver.
  • We shouldn’t almost hit pedestrians because that wouldn’t make anyone happy.
  • We should consider speed as well when it comes to this distance, people are much more comfortable with a slower car being closer to them as opposed to a car going 60 miles an hour.

What happens if the user tries to enable cruise control while the brake by wire is enabled?

  • We can assume that these systems will cooperate and not cause unexpected problems.
  • We can also assume that collision avoidance will take priority when both systems are active.

Should the PCAS work for random objects like branches and whatnot?

  • For this project, it’ll only work with pedestrians.
  • Is something to consider in the writeup for potential things to add in the future.

What happens when a pedestrian stops within the vehicle's path?

  • We’ll need to include this scenario within the system.
  • How should we act when this scenario is detected, we could potentially trigger driver override.
  • If we choose the option to let the driver override what does this look like inside and outside of the vehicle and would the vehicle come to a stop before this override happens or could the driver take over on the fly?

How should the system account for malfunctions or packet loss?

  • This is a failure mode for us to take into account in our design.

What steps should we take if other systems aren’t working properly in the vehicle?

  • We can assume other systems are working.
  • We can also assume that all systems are working in cooperation with the PCAS and that the PCAS is taking high priority over other systems within the vehicle.

What quality breaks can we assume for the system?

  • For our simulations, we should be using realistic breaks.
  • Follow-up question: Do we need to consider weather conditions with breaks?
  • - No this is out of scope but is a good point to consider in the write-up.

Is it safe to assume we’ll have maximum deceleration or acceleration in an instant?

  • This is defined in the document within the section on the brake by wire actuation.

Is it safe to assume we’ll have maximum deceleration or acceleration in an instant?

  • This is defined in the document within the section on the brake by wire actuation.

How quickly should the car get back up to speed after a collision is avoided?

  • What we determine as safe and comfortable to do so.

What if the system is offline?

  • This would increase the security of the system and wouldn’t hurt the useability of the system.
  • Could be a step to make a more secure system, but would open up other aspects we would have to consider such as hotwiring the system.

What is the confident detection range?

  • The range where we can assume that the pedestrian will be detected by the radar.

Does the system have any accessibility through Bluetooth and wifi?

  • It is completely up to us.
  • Typically, the system tends to be local and hardwired in.

Any specific platform the solution should be developed on?

  • This will be up to us to decide.

Should security just be up to us if it's not wireless? Then how would it be secure in terms of the system being wired?

  • Part of it could be there are some wireless connections to the vehicle and how it is independent of those systems because if it isn’t independent it could be accessed.
  • We’ll have to take time to separate the system from other parts of the car.
  • The technology isn’t defined clearly so we have some choices to make.

Any simulation software we could have access to?

  • No, we'll have to find our own.

What makes this system stand out?

  • The safety-critical aspect of the system makes it stand out, safety is the highest priority of this system.

Any specific deliverables from us as contributors?

  • Nothing specific has been defined yet.
  • But visuals are always nice and helpful, they are the most impactful way to show what is going on.
